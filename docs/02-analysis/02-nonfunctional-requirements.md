# Нефункциональные требования проекта SnapMatch

## Безопасность

- **NFR-001**: Авторизация пользователей осуществляется через Keycloak по протоколу OAuth 2.0 (authorization_code grant).
- **NFR-002**: Все запросы проходят по HTTPS, TLS v1.2+ обязателен.
- **NFR-003**: Конфигурация Keycloak хранится в зашифрованном виде в Kubernetes Secrets.


## Производительность и масштабируемость

- **NFR-004**: Соискатель должен получать вердикт о статусе резюме при в течение 5 минут нагрузке до
100 одновременных пользователей
- **NFR-005**: Масштабирование осуществляется автоматически через Horizontal Pod Autoscaler
- **NFR-006**: Взаимодействие с API LLM должно быть веделено в отдельный микросервис
- **NFR-007**: В качестве LLM используются модели Gigachat-2-Pro или Gigachat-2-Max


## Хранение файлов (MinIO)

- **NFR-008**: Все загружаемые пользователями файлы сохраняются в объектное хранилище MinIO 
в режиме single-node, без репликации и версионирования
- **NFR-008**: В БД должны храниться ссылки на файлыс резюме в MinIO
- **NFR-009**: Размер файла с резюме не должен превышать 10 МБ
- **NFR-010** Файл доступен для чтения сервису обработки резюме.

## Мониторинг и логирование

- **NFR-013**: Все контейнеры логируют в stdout/stderr, логи агрегируются в Fluent Bit и отправляются в OpenSearch.
